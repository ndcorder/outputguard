# outputguard 2.0 Plan

## Goal

Make the Python package match the JavaScript 2.0 direction: a production structured-output toolkit with format-aware parsing, provider-agnostic guarded generation, batch diagnostics, and CLI support.

## Implemented Scope

### 1. Format-Aware Core

JSON remains the default. Public validation, repair, parse, retry-prompt, `OutputGuard`, reports, exceptions, and CLI commands accept `format` / `--input-format` for:

- `json`
- `yaml` / `yml`
- `toml`
- `python` / `python-literal` / `literal`
- `auto`
- `forced-json-off`

Generic repairs such as fence stripping preserve non-JSON syntax where possible.

### 2. Guarded Generation

`guarded_generate()` wraps any synchronous provider function:

- Calls `generate(prompt, context)` with attempt metadata.
- Validates and optionally repairs each response.
- Builds targeted retry prompts from schema errors.
- Returns every attempt for observability.
- Supports strict `repair=False` mode.
- Supports `throw_on_failure=True` via `GuardedGenerationError`.

`guarded_generate_async()` provides the same flow for async provider clients.

### 3. Batch Processing

`validate_batch()` and `repair_batch()` support corpus checks, eval fixtures, and production log audits:

- Preserve per-item index and input.
- Return full per-item results.
- Summarize total, valid, invalid, repaired, parse failures, schema failures, success rate, formats, and strategy counts.
- CLI `outputguard batch` validates a JSON array of output strings and returns JSON or text summaries.

### 4. Documentation And Metadata

The README documents the 2.0 APIs, supported formats, batch CLI, strategy count, dependency set, and test count. Package metadata is bumped to `2.0.0` and includes keywords for guardrails, batch processing, Python literals, and forced-JSON-off output.

## Non-Goals

- No provider SDK dependencies.
- No schema generation from Python types or Pydantic models.
- No release automation changes.
- No new real-model API sweep. The batch API supports fixture and eval workflows without requiring credentials in the test suite.

## Verification Contract

2.0 is considered complete when:

- Focused red/green tests cover guarded generation and batch helpers.
- Full pytest suite passes.
- Ruff lint passes.
- Ruff format check passes.
- Mypy passes for `outputguard`.
- `git diff --check` passes.
- README and package metadata describe the implemented APIs accurately.
